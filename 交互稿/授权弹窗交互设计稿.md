# 会员福利授权弹窗 — 交互设计稿

> 版本：v1.0 | 日期：2026-05-26 | 作者：产品团队

---

## 一、功能概述

用户在美柚App领取会员福利后，通过授权弹窗同意将手机号提供给对应服务商，由服务商主动联系用户完成福利核销。

**核心目标**：降低用户核销行动成本 → 提升福利核销率。

---

## 二、触发时机

| 场景 | 触发条件 | 优先级 |
|------|----------|--------|
| **场景A：领福利后立即弹（主路径）** | 用户点击"立即领取" → 领取成功 → 自动弹出授权弹窗 | P0 |
| **场景B：福利详情页手动触发** | 用户在"我的福利"列表点击福利 → 详情页内有"授权服务商联系我"入口 | P1 |
| **场景C：推送提醒后触发** | 用户收到"福利即将过期"Push → 点击进入详情页 → 展示授权入口 | P2 |

> 场景A弹窗频率控制：同一用户对同一服务商，7天内最多弹1次；已被拒绝后，7天内不再自动弹。

## 三、完整交互流程图

`mermaid
flowchart TD
    A[用户点击领取福利] --> B{领取成功?}
    B -->|失败| B1[提示领取失败, 可重试]
    B -->|成功| C{该福利类型是否需要<br/>服务商联系核销?}
    C -->|不需要| D[展示领取成功页<br/>引导用户自行核销]
    C -->|需要| E{该用户是否已授权<br/>过该服务商?}
    E -->|是 & 未过期| F[跳过弹窗<br/>服务商侧已有线索]
    E -->|否 or 已过期| G[弹出授权弹窗]
    G --> H{用户操作}
    H -->|同意授权| I[调用授权接口]
    I -->|成功| J[Toast: 授权成功<br/>服务商将在24h内联系您]
    I -->|失败| K[Toast: 网络异常请重试<br/>福利详情页可再次授权]
    H -->|暂不授权| L[关闭弹窗<br/>进入福利详情页]
    L --> M[详情页底部展示<br/>授权入口卡片]
    M --> G
    J --> N[福利状态更新为<br/>已授权·待联系]
`

│                                     │
│  ┌─────────┐    ┌───────────────┐   │
│  │ 暂不授权  │    │   ▶ 同意授权   │   │  ← 同意按钮为主按钮(高亮)
│  └─────────┘    └───────────────┘   │
│                                     │
└─────────────────────────────────────┘
`

**交互说明：**

| 元素 | 说明 |
|------|------|
| 主标题 | 传达价值（帮你快速核销），而非索取（我要你的电话） |
| 手机号 | 展示脱敏手机号，增强信任感 |
| 信息框 | 灰色背景卡片，明确告知共享范围和有效期 |
| 勾选框 | **必须单独勾选**，不能默认打勾，符合《个保法》单独同意要求 |
| 同意按钮 | 勾选框未勾选时置灰不可点；勾选后高亮可点 |
| 暂不授权 | 文字按钮，低视觉权重，点击后关闭弹窗 |
| 关闭手势 | 支持点击蒙层区域关闭（视为"暂不授权"） |

## 五、撤回授权流程

`mermaid
flowchart LR
    A[用户点击撤回授权] --> B[二次确认弹窗]
    B --> C{确认撤回?}
    C -->|确认| D[调用撤回接口]
    D --> E[服务商侧删除用户信息]
    E --> F[用户侧更新为已撤回]
    F --> G[Toast: 已撤回, 服务商不再联系您]
    C -->|取消| H[关闭弹窗]
`

## 六、异常态 & 边界情况

| 场景 | 处理方式 |
|------|----------|
| 网络异常导致授权失败 | Toast提示"网络异常，请稍后重试"，不关闭弹窗，用户可重试 |
| 服务商已下线/不再合作 | 不展示授权弹窗，仅展示自助核销路径 |
| 用户已授权但7天后过期 | 福利详情页展示"授权已过期，重新授权 →"入口 |
| 用户已对同一服务商授权过（不同福利） | 复用已有授权，不重复弹窗；展示"已授权，该服务商将一并处理" |
| 用户手机号变更 | 美柚端检测手机号变更后，已有授权自动失效；新授权使用新手机号 |
| 服务商侧联系超时（24h未联系） | 福利详情页展示"服务商暂未联系，可重新授权或自行核销" |
| 用户投诉/标记骚扰 | 触发熔断，暂停该服务商新授权获取，人工介入 |

弹窗中需要动态替换的字段：

| 变量 | 来源 | 示例 |
|------|------|------|
| {服务商名称} | 福利配置后台 | 固德口腔 |
| {福利名称} | 福利配置后台 | 超声波洗牙 |
| {脱敏手机号} | 用户账号信息 | 138****6789 |
| {授权天数} | 系统配置（建议15天） | 15天 |
| {联系时效} | 服务商SLA配置 | 24小时内 |

## 八、埋点建议

| 事件名 | 说明 |
|--------|------|
| uth_popup_show | 弹窗曝光，区分场景A/B/C |

| uth_success | 授权接口成功 |
| uth_fail | 授权接口失败（含错误码） |

| uth_detail_entry_click | 从福利详情页点击授权入口 |

| enefit_redeem_success | 福利核销成功（含核销方式：服务商联系/自助） |

## 九、与后台接口交互

`
POST /api/member/benefit/authorize

Request:
{
  "benefit_id": "12345",        // 福利ID
  "merchant_id": "67890",       // 服务商ID
  "agree_protocol": true        // 是否同意协议
}

Response (200):
{
  "code": 0,
  "data": {
    "auth_id": "auth_001",
    "phone_masked": "138****6789",
    "merchant_name": "固德口腔",
    "expire_time": "2026-06-10 14:30:00"
  }
}
`

**撤回接口**：
`
POST /api/member/benefit/revoke

Request: { "auth_id": "auth_001" }
Response: { "code": 0, "message": "已撤回授权" }
`

## 十、设计决策记录

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 弹窗 vs 半屏面板 | **居中弹窗** | 授权类操作需要用户注意力集中，弹窗比底部面板更正式、更合规 |
| 授权粒度 | **单服务商+单福利** | 合规要求明确告知接收方，合并授权有风险 |
| 有效期 | **15天** | 覆盖2个周末，给用户足够决策时间；太短则服务商来不及联系 |
| 弹窗抑制 | **被拒绝后7天不再弹** | 避免骚扰用户，平衡转化率与体验 |
| 授权管理入口 | **设置-隐私管理** | 符合用户心智模型，撤回授权的入口应在隐私/设置中 |
