# 美柚中后台 PRD 模板

## 标题格式
```
{谁（运营人员/产品人员）}可以通过{什么功能}获得{什么价值（提升效率/实现业务核心指标的配置）}
```

示例：运营人员可以通过会员标签批量配置功能提升人群定向效率

---

## 一、需求背景（20%）

### 1.1 业务大背景（5%）
对齐团队 OKR（Object and Key Results），需求是为了实现团队哪个大 O（战略目标）而努力的，Review 需求方向不偏离组织战略方向。

### 1.2 业务子背景（5%）
通过用户反馈 & 用户 case 分析、体验感受等维度，通过一两句话概括做这个需求的业务背景。

- 包含用户反馈、实际使用中发现的负向 case（在中后台产品中，"用户"指业务方）
- 商业化需求则论述来自客户方的信息，包括客户背景、商业化诉求

### 1.3 现状判断及问题（10%）
围绕系统现状、业务现状使用分析，判断满足业务需求的核心点。

| 现状 | 问题判断 | 历史需求或复盘文档 | 解决方案 |
|------|----------|-------------------|----------|
|      |          |                   |          |

---

## 二、项目目标（5%）

### 2.1 目标描述（5%）
本次迭代的价值，如：运营可以灵活配置某功能，或可以通过某功能提升人效，或降低资源投入成本。

### 2.2 迭代节奏（非必填）
是否需要拆版本或分阶段实现，每个阶段实现的功能分别有哪些。

### 2.3 风险预判（非必填）
是否存在风险，如：可能造成延期的技术难点突破，是否可被妥协、对于需求方是否会有负向影响，影响范围。

---

## 三、需求方案（80%）

### 3.1 名词定义（非必填）
在本需求中，新增的名词，统一"名词"口径，阐述清楚其定义。

| 名词 | 定义 |
|------|------|
|      |      |

### 3.2 E-R图（非必填）
**必须使用中文命名**：实体名、属性名、关系名均须使用中文（如"广告分组"、"广告源"、"包含"），不得使用英文或拼音。

新增字段通过 E-R 图表达信息结构，表示实体、属性，以及它们之间的联系。如：用户和商品间的关系是下单，表示为"下单"。下单这个行为用菱形括起来，下单后就会有订单编号、下单时间等属性，用圆角矩形括起来，并和"下单"行为连接在一起。1 个用户可以下 M 个订单，则将数字"1"和"M"写在直线上。

### 3.3 产品结构图（10%）
系统的结构图（可以用脑图呈现），包含系统中包含的：模块、功能。*"商业化产品"统分时，不涉及。*

### 3.4 产品流程图（10%）
多角色/跨职能/跨系统流程图，即"泳道图"。只涉及一个角色或系统并没有交互，不必画泳道图，画普通流程图即可。只画到业务层面，不用详细到技术层面。

### 3.5 原型图（非必填）
可视化后台需求，需提供后台原型图。

### 3.6 需求说明（TAPD）（60%）
需求列表包含功能模块、功能点描述、优先级。详细说明包含功能点、功能详细说明、补充说明等。

| 功能模块 | 功能点描述 | 优先级 |
|----------|-----------|--------|
|          |           |        |

**详细说明：**

**功能点：[名称]**
- 功能详细说明：
- 补充说明：

### 3.7 对协同方需求（运营、编辑、商务等）
如：功能需求中涉及到需运营配置的内容策略、落地页；需要编辑提供的文案；需要商务侧提供外部合作方在项目周期内联调支持等。

## 元数据字段

| 字段 | 说明 |
|------|------|
| 状态 | 规划中 / 研发定级 / 需求价值评审 / 挂起 |
| 创建模板 | 中后台需求模版 |
| 分类 | 业务分类 |
| 业务 | 所属业务线 |
| 迭代 | 迭代版本 |
| 处理人 | 需求负责人 |
| 优先级 | High / Middle / Low |
| 需求类型 | 基础建设 / 新功能 / 优化 |
| 需求难度 | A 全新能力 / B 现有能力迭代 / C 简单配置 |
| 技术等级 | -- |
| 预估工时 | 人天 |
